
# Business Logic Layer (BLL)
**Project: Automated CI/CD Pipeline with Intelligent Failure Recovery**

*Note: As this project is a React-based frontend demo, the Python EC2 backend prescribed in the original  system architecture is simulated using JavaScript modules in the `src/bll/` directory. These module files act precisely as the Business Logic Layer, handling all verification, computation, and transformation, decoupled from the React UI components.*

---

## Q1. Core Functional BLL Modules & UI Interaction 

The Business Logic Layer sits between the `src/pages/` (Presentation Layer) and `src/data/mockData.js` (Simulated Database Layer). The UI components contain strictly zero calculation or manipulation logic; they only execute BLL functions and render the pre-formatted outputs.

Below are the 7 core modules identified from the Software Requirements Specification (SRS), mapping to their respective frontend interaction:

### 1. `CICDPipeline.js` (Main Orchestrator — SRS FR1-4)
- **Role:** Handles manual triggers, pipeline creation, build numbering, and live progress state.
- **UI Interaction:** The `RunModal` form in `App.jsx` submits user inputs parameters to `startPipeline()`. If successful, the BLL generates a new pipeline record. The `Dashboard.jsx` constantly pulls `getStatus()` to animate the active pipeline visualizing its progression through 8 stages.
- **UI Interaction:** The "Stop Pipeline" mechanism aborts the running BLL state, forcing a failed badge.

### 2. `BuildManager.js` (Build Engine — SRS FR2)
- **Role:** Computes build duration averages, calculates success rates, and manages artifact lifecycles.
- **UI Interaction:** `Builds.jsx` fetches `getBuildStats()` to dynamically update Top-bar KPI cards. The function applies success rate thresholds to return a specific alert color (ok = cyan, warning = yellow, critical = red).

### 3. `TestManager.js` (Testing Engine — SRS FR3)
- **Role:** Simulates the continuous testing phase and streams formatted console logs.
- **UI Interaction:** `Logs.jsx` triggers `simulateTestStep()`, pulling color-coded log lines dynamically generated by the BLL based on testing status strings (INFO, WARN, ERROR) into a live terminal view.

### 4. `DeploymentManager.js` (Target Sync — SRS FR4)
- **Role:** Groups historical pipelines by target execution environment (e.g., Development vs. Production).
- **UI Interaction:** `Pipelines.jsx` uses `getDeploymentSummaryByEnvironment()` to render the specific success margin badges located on the right sidebar table.

### 5. `FailureAnalyzer.js` (Failure Classification — SRS FR5-6)
- **Role:** The core "intelligent" module that uses keyword pattern matching on raw error logs to classify failures (e.g., "timeout" = Network Error) and enforce rollback checks.
- **UI Interaction:** `Failures.jsx` pulls `getFailureDistribution()` to render percentage-based progress bars. Clicking "Rollback" on the UI triggers `rollbackBuild()`, executing BLL checks for manual-rollback eligibility.

### 6. `RecoveryManager.js` (Auto-Recovery Policy — SRS FR7-8)
- **Role:** Manages the retry system, ensuring auto-recovery loops are capped and fallback actions are validated.
- **UI Interaction:** `Recovery.jsx` submits slider values to `saveRecoveryConfig()`. The configurations directly influence the BLL's `decideRecoveryAction()` decision engine tree.

### 7. `ReportGenerator.js` (Analytics Processing — SRS FR9)
- **Role:** Compiles all historical mock data into analytics shapes compatible with charts and generates downloadable file streams.
- **UI Interaction:** `Reports.jsx` submits a date range to `generateReport()`, which the BLL validates and returns as a JSON string, converted by the UI into a downloadable file payload (`Blob`).

## Q2. Business Rules, Validation, and Data Transformation 

### Q2-A: Business Rules Implementation
Business rules dictate operational policies. They are centrally located and rigidly enforced within the corresponding `src/bll/` modules.

1. **Trigger Policies (FR1):**
    - Pipelines cannot initiate unless triggered on a strictly whitelisted Git branch (`main`, `dev`, `hotfix`, etc.).
    - Target deployment environments must be valid string constants (`development`, `staging`, `production`).
2. **Sequential Progression Rules (FR4):**
    - The deployment manager prohibits a `production` pipeline run unless a corresponding successful `staging` pipeline already exists. 
3. **Quality Gates (FR2/FR3):**
    - Build stages fire severe CRITICAL warnings if historical success rates drop below a strictly mathematical 70% threshold.
    - Test execution fails entirely if the calculated testing pass rate falls beneath an 80% boundary.
4. **Intelligent Escalation & Override Rules (FR6/FR7):**
    - Failures classified by the BLL via keywords as "Deploy Errors" bypass the retry system and are forcefully queued for Manual Intervention.
    - An auto-retry system tracks attempts. Exceeding a maximum attempt threshold (e.g., 3 Retries) automatically triggers an irreversible system rollout rollback fallback loop.
    - Administrative Users hold ultimate override capabilities. Clicking "Stop Pipeline" or explicitly initiating a Rollback immediately terminates any underlying BLL pipeline loops.

### Q2-B: Validation Logic
The BLL protects the data layer by forcing all UI inputs through a structured Validation pattern. The system accumulates errors into a collection array instead of halting on the very first exception.

**Validation Response Pattern:**
```javascript
export function validatePipelineTrigger(data) {
  const errors = [];
  if (!isValidURL(data.url)) errors.push('URL must be https or git@');
  if (!VALID_ENV.includes(data.env)) errors.push('Invalid Env target');
  
  return { isValid: errors.length === 0, errors }; // UI iterates this array to spawn Toasts
}
```

**Key BLL Validations Executed:**
| Module | Validated Action | Criteria Evaluated |
|---|---|---|
| `CICDPipeline` | **Pipeline Triggering** | Repository URL formats (https/git@), Branch Whitelist approval, Empty character fields. |
| `RecoveryManager`| **Policy Updates**   | Numeric limits strictly between 10s and 300s. Fallback modes matching specific allowed strings. |
| `ReportGenerator`| **Metrics Reporting**| Chronological Date comparisons (End Date > Start Date), Maximum history retrieval boundaries (Max 365 Days). |
| `FailureAnalyzer`| **Manual Rollback** | Must target an explicitly "FAILED" build. Skips rollback processing entirely if the build error type allows auto-retries. |

### Q2-C: Data Transformation
The UI's Presentation Layer (`React`) is incapable of performing algorithmic operations. The BLL connects to raw rows in `mockData.js` and performs complex data transformations, returning structured components optimized for visual consumption.

1. **Time Value Calculations:**
   - *Raw Input:* `[{duration: "4m 22s"}, {duration: "3m 08s"}]`
   - *BLL Processing:* Regex parsing to individual seconds integers (`262s + 188s`), computes mathematical average, formats modulo string back to standard output.
   - *Transformed Output:* `"3m 45s"` string printed cleanly inside a React KPI card.
2. **Analytics Aggregation for Charts:**
   - *Raw Input:* Array of unordered chronological pipeline objects.
   - *BLL Processing:* Groups entities dynamically by specific Javascript Date "Day-of-Week" indices, incrementing counters for status types.
   - *Transformed Output:* Array structure `[{day: 'Mon', passed: 3, failed: 1}, ...]` perfectly designed to be piped directly into `Recharts.js` React components without alteration.
3. **Error Identification Generation:**
   - *Raw Input:* Messy string trace log `"Connection Refused: Timeout elapsed"`
   - *BLL Processing:* Executes recursive inclusion keyword testing checking mapped definitions.
   - *Transformed Output:* Outputs a structural Object `{ type: "Network Timeout", color: "#eab308" }` assigning explicit failure categorizations and assigning React Hex variables directly for the UI representation.
4. **Resolved Failure Cleanup:**
   - *Raw Input:* Static mock database objects marking failed builds.
   - *BLL Processing:* When users successfully engage "Retry" or "Rollback", BLL mutates status state tracking, and immediately strips/filters those entries utilizing Array Manipulation from the `getEnrichedFailures()` active history payload so they visibly disappear resolving the issue smoothly in the UI.
